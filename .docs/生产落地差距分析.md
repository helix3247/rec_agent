# 项目生产落地差距分析与规划

本文档基于 `d:\Project\rec_agent\.docs\开发计划.md` 完成阶段五（高阶推理能力）后的项目状态，分析距离生产级落地（Production Ready）的主要差距，并提出规划建议。

---

## 1. 核心功能现状 (Status Quo)

截至目前，后端核心 Agent 逻辑已实现 **100%** 覆盖，具备以下高阶能力：

- ✅ **意图识别与槽位填充** (`IntentParser`): 准确识别搜索、穿搭、问答、规划等意图。
- ✅ **多轮澄清对话** (`DialogFlow`): 遇到信息缺失（如预算、用途）时主动追问。
- ✅ **复杂任务规划** (`Planner`): 将 "西藏旅游全套装备" 拆解为 "穿搭+装备+防护" 分步执行。
- ✅ **自我反思与修正** (`Reflector`): 自动检测空结果或不合理需求（如"200元买全新单反"），并进行自我修正或引导。
- ✅ **个性化排序** (`Personalization`): 基于用户画像重排序推荐结果。
- ✅ **RAG 问答**: 基于 Milvus 知识库回答商品细节问题。

---

## 2. 生产落地差距分析 (Gap Analysis)

虽然核心大脑已就绪，但要达到生产级标准（高可用、低延迟、可观测、安全），还存在以下关键差距（按优先级排序）：

### 2.1 真实数据接入与实时管道 (Data Pipeline)
目前使用的是 Mock 数据或一次性脚本导入的数据。生产环境需要建立实时数据流：

- **商品实时同步**:
    - **现状**: 依赖 `scripts/sync_to_vector_db.py` 手动同步。
    - **目标**: 监听业务数据库 Binlog（如 Canal/Debezium），商品上架/下架/改价秒级同步至 Elasticsearch。
- **用户行为流 (User Events)**:
    - **现状**: 画像数据静态存储在 MySQL。
    - **目标**: 用户的实时点击/加购/下单行为需通过消息队列（Kafka/Pulsar）写入画像系统，实时更新 `db.get_user_profile` 结果，实现"刚才看了鞋子，现在立刻推荐袜子"。
- **知识库自动化更新**:
    - **现状**: 手动运行脚本向量化。
    - **目标**: 新发布的商品说明书、UGC 评论需自动化触发 ETL 流程，清洗后向量化写入 Milvus。

### 2.2 系统稳定性与高可用 (Reliability & HA)
目前 `graph.py` 是单机内存运行，存在单点风险。

- **状态持久化 (State Persistence)**:
    - **现状**: LangGraph 状态存储在内存中，服务重启会导致会话丢失。
    - **目标**: 使用 Redis 或 Postgres 持久化 Checkpointer，支持服务重启后无缝恢复对话 (`thread_id` 关联)。
- **熔断与降级 (Circuit Breaking)**:
    - **现状**: 仅实现了 LLM 调用的 `try-except` 切换 Fallback 模型。
    - **目标**: 对 ES、Milvus、MySQL 的调用增加熔断机制（如使用 `pybreaker`），防止下游故障导致 Agent 线程阻塞雪崩。
- **并发与限流**:
    - **现状**: 无并发控制。
    - **目标**: 针对 `POST /chat` 接口实施令牌桶限流，防止恶意刷量消耗 Token 预算；对 Agent 内部的并发子任务（如 Planner 并发调用 Shopping）设置信号量限制。

### 2.3 生产级可观测性 (Observability)
目前仅有简单的日志打印，难以排查复杂链路问题。

- **全链路追踪 (Tracing)**:
    - **现状**: 仅有 Loguru 日志。
    - **目标**: 接入 LangSmith 或 Jaeger/OpenTelemetry，追踪每一条 Query 在 Agent 链路中的完整流转（Intent -> Dispatcher -> Shopping -> Reflector），可视化 Token 消耗与耗时瓶颈。
- **业务指标监控 (Metrics)**:
    - **现状**: 无。
    - **目标**: 接入 Prometheus/Grafana，监控关键指标：
        - LLM 响应时间 (P95/P99)
        - 意图识别准确率 (人工抽检或模型打分)
        - 推荐结果空置率 (Reflector 触发率)
        - 任务完成率 (Goal Success Rate)

### 2.4 前端交互体验 (Frontend UX)
目前仅提供同步 REST API，用户体验有待提升。

- **流式输出 (Streaming)**:
    - **现状**: 等待所有 Agent 执行完一次性返回 JSON。
    - **目标**: 支持 SSE (Server-Sent Events) 流式输出。
        - **UI 呈现**: 让用户先看到 "正在思考任务拆解..." -> "正在检索背包..." -> "正在生成建议..."，减少等待焦虑。
        - **打字机效果**: LLM 生成的长文本逐字显示。
- **多模态交互**:
    - **现状**: 纯文本交互。
    - **目标**: 支持用户发送图片（"我要买这件衣服的搭配"），利用多模态大模型（如 GPT-4o-mini / Gemini-Flash）解析图片意图。

### 2.5 安全与合规 (Security & Compliance)

- **Prompt 注入防护**:
    - **目标**: 增加 Guardrails 层，防止用户通过恶意指令（如"忽略所有规则"）绕过导购逻辑或套取 System Prompt。
- **敏感数据脱敏**:
    - **目标**: 日志中自动化脱敏用户 PII 信息（手机号、地址）。
- **内容风控**:
    - **目标**: LLM 生成的回答需经过敏感词过滤，确保不输出违规内容。

## 3. 下一步规划建议 (Roadmap Recommendation)

建议按照 **P0 (核心体验)** -> **P1 (稳定性)** -> **P2 (运营能力)** 的顺序推进：

| 优先级 | 任务模块 | 关键动作 | 预期价值 |
| :--- | :--- | :--- | :--- |
| **P0** | **前端交互** | 实现 SSE 流式接口 + 简单的 Chat UI (React/Vue) | 让系统"可演示、可交付"，大幅提升用户体验 |
| **P0** | **真实数据** | 搭建基础的 ES/Milvus 增量同步脚本 | 摆脱 Mock 数据，让推荐结果真实可信 |
| **P1** | **持久化** | 引入 Redis Checkpointer | 支持多轮对话的长久记忆，服务重启不丢状态 |
| **P1** | **监控** | 接入 LangSmith 或基础 Prometheus 监控 | 具备排查线上问题的能力，量化 Token 成本 |
| **P2** | **评测闭环** | 建立 Golden Set 自动化回归测试 | 保证迭代不劣化，支持 Prompt A/B 测试 |
| **P2** | **多模态** | 增加图片上传与解析能力 | 拓展"以图搜图"、"穿搭点评"等高价值场景 |
