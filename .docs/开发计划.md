# 电商推荐 Agent 项目详细开发计划

本文档基于 [技术方案](./技术方案.md) 编写，采用 **"Walking Skeleton" (行走骨架)** 策略，将 Multi-Agent 系统的开发过程拆解为可执行、可测试的详细子任务。

> **开发范围说明**：本开发计划仅覆盖**后端 API 与 Agent 系统**的开发。前端界面不在当前版本范围内，所有功能通过 `POST /chat` RESTful API 对外暴露，可使用 Postman / curl / Swagger UI 进行测试与演示。如需前端展示，建议后续独立规划前端开发阶段。

---

## 1. 阶段一：环境准备与基础设施 (Infrastructure & Setup)

**目标**：完成开发环境配置，跑通 Hello World，确保 LLM 接口与数据层可用。

**阶段验收**：LLM 调用成功、Docker 全部容器健康、数据初始化脚本跑通。

### 1.1 环境配置
- [ ] **Python 虚拟环境搭建**
    - **内容**：使用 Conda 创建隔离环境。
    - **验证**：`conda activate rec_agent` 成功，`python --version` 输出正确 (3.10+)。
- [x] **依赖管理**
    - **内容**：编写 `requirements.txt`（建议固定主版本），至少包含：
        - 框架：`langchain`, `langgraph`, `fastapi`, `uvicorn`, `pydantic`, `pydantic-settings`
        - LLM：`openai`（或兼容 DeepSeek 的 OpenAI API SDK）
        - 工具与观测：`loguru`, `python-dotenv`, `httpx`, `langsmith`
        - 数据层：`pymilvus`, `elasticsearch`, `redis`, `aiomysql`（或 `pymysql`）
        - 评测：`ragas`
    - **验证**：`pip install -r requirements.txt` 无报错。
- [x] **配置管理**
    - **内容**：
        - 创建 `.env.example` 模板文件。
        - 创建 `app/core/config.py`，使用 `pydantic-settings` 加载环境变量。
    - **要求**：严禁将真实 Key 提交到 Git；配置类需包含 `LLM_API_KEY` (DeepSeek)、`FALLBACK_LLM_API_KEY` (OpenAI) 和 `LANGSMITH_API_KEY`。
    - **验证**：运行简单的 Python 脚本，能正确打印出配置项的值（注意脱敏）。

### 1.2 项目结构搭建
- [x] **目录初始化**
    - **内容**：建立 Multi-Agent 目录结构：
        - `app/core` (配置、日志、LLM 客户端)
        - `app/agents` (智能体实现：dispatcher, intent, shopping, rag, toolcall, dialog, fallback, monitor, outfit, response_formatter)
        - `app/tools` (工具函数：search, db, memory, personalization)
        - `app/prompts` (各 Agent 的 System Prompt)
        - `app/api` (FastAPI 路由)
        - `app/models` (Pydantic 数据模型)
        - `scripts` (运维与测试脚本)
        - `tests` (单元测试与评测)
    - **要求**：每个目录下包含 `__init__.py`。
    - **验证**：目录结构清晰，无冗余文件。

### 1.3 数据层本地环境 (Docker Compose)
- [x] **Docker/Compose 一键启动**
    - **内容**：提供 `docker-compose.yml`（或 `compose.yaml`），包含 MySQL、Elasticsearch、Milvus(Standalone)、Redis（以及依赖组件如 etcd/minio）。
    - **要求**：服务间网络互通、端口清晰、数据卷持久化、基础健康检查。
    - **验证**：`docker compose up -d` 后，全部容器健康；本机能连通 MySQL/ES/Redis/Milvus。
- [x] **样例数据与索引初始化** *(已完成)*
    - **内容**：已提供以下脚本（位于 `scripts/` 目录）：
        - `scripts/config.py` — 统一配置管理（MySQL/ES/Milvus/Embedding 连接参数、数据规模）。
        - `scripts/generate_mock_data.py` — 生成 mock 数据（用户、商品、评论），输出到 `data/mock_data.json`。
        - `scripts/init_mysql.py` — 初始化 MySQL 表结构并批量写入用户/商品数据。
        - `scripts/sync_to_vector_db.py` — 将商品同步到 ES（Hybrid Search 索引）、评论/文档同步到 Milvus（RAG 知识库）。
    - **验证**：依次执行 `generate_mock_data.py` -> `init_mysql.py` -> `sync_to_vector_db.py` 后，能通过工具函数查询到商品、评论/FAQ chunk。

### 1.4 LLM 联通与基础设施
- [x] **多模型客户端封装**
    - **内容**：在 `app/core/llm.py` 中封装 LLM Client 工厂。
    - **要求**：实现 `get_llm(model_type)`，支持获取 "primary" (DeepSeek) 和 "fallback" (OpenAI) 两种客户端。
    - **验证**：编写 `scripts/test_llm.py`，分别测试主模型和备用模型的连通性。
- [x] **基础容错封装（LLM 调用层）**
    - **内容**：在 `app/core/llm.py` 中为 LLM 调用增加 `try/except` 降级逻辑：主模型失败时自动切换到 fallback 模型。
    - **要求**：记录降级日志（模型名、错误原因、是否降级成功）。
    - **验证**：故意配置错误的主模型 Key，验证调用是否自动降级到 fallback 并返回正常结果。
- [x] **日志系统配置**
    - **内容**：配置 `loguru`，定义日志格式（包含时间、级别、TraceID、AgentName）。
    - **要求**：日志同时输出到控制台和 `logs/app.log` 文件；支持按天轮转。
    - **验证**：运行测试脚本，检查控制台和日志文件中是否有符合格式的日志输出。

## 2. 阶段二：行走骨架 (The Walking Skeleton)

**目标**：实现一个**没有大脑、没有数据库**，但流程完整的 Multi-Agent 骨架。数据在节点间流转，接口可访问。

**阶段验收**：`curl POST /chat` 能收到 Mock 响应，Graph 图结构可视化正确。

### 2.1 状态定义 (State Definition)
- [x] **AgentState 定义**
    - **内容**：在 `app/state.py` 中定义全局 `AgentState` TypedDict。
    - **要求**：包含 `messages` (对话历史), `user_intent` (意图), `current_agent` (当前执勤Agent), `task_status` (任务状态), `trace_id`。
        - **补充要求**：为后续多轮/个性化/收藏夹预留字段：`thread_id`, `user_id`, `selected_product_id`（或 `focus_product_id`）, `slots`（预算/用途/风格等槽位）。
    - **验证**：编写单元测试，验证 State 可以被正确初始化和修改。

### 2.2 核心 Agent Mock 实现
- [x] **IntentParserAgent (Mock)**
    - **内容**：实现 `app/agents/intent_parser.py`，返回固定 JSON `{"intent": "search_product", "params": {}}`。
- [x] **TaskDispatcherAgent (Mock)**
    - **内容**：实现 `app/agents/dispatcher.py`，根据 Intent 路由到下一个节点。
- [x] **ShoppingAgent (Mock)**
    - **内容**：实现 `app/agents/shopping.py`，返回固定文本 "这是 Mock 的推荐结果：Sony A7M4"。
- [x] **ResponseFormatter (Mock)**
    - **内容**：实现 `app/agents/response_formatter.py`，透传上游输出，拼装标准 Response Body 结构。
- [x] **MonitorAgent (Mock)**
    - **内容**：实现 `app/agents/monitor.py`，简单打印日志 "Request completed"。

### 2.3 图构建 (Graph Construction)
- [x] **LangGraph 组装**
    - **内容**：在 `app/graph.py` 中使用 `StateGraph` 串联上述节点。
    - **要求**：
        - 定义 `START` -> `IntentParser` -> `Dispatcher` 的边。
        - 定义 `Dispatcher` 的条件边 (Conditional Edges) 分发给各 Specialist Agent。
        - 所有 Specialist Agent 执行完 -> `ResponseFormatter` -> `Monitor` -> `END`。
        - 编译图 (`graph.compile()`)。
    - **验证**：使用 `graph.get_graph().draw_ascii()` 打印图结构，确认连接正确。

### 2.4 API 接口层
- [x] **FastAPI 路由**
    - **内容**：编写 `app/api/endpoints/chat.py`，定义 `POST /chat` 接口。
    - **要求**：
        - Request Body: `{"query":"...","thread_id":"...","user_id":"...","selected_product_id":"..."}`（`thread_id` / `user_id` / `selected_product_id` 可选；无 `thread_id` 则创建新会话，有则续聊）。
        - Response Body: `{"response":"...","trace_id":"...","thread_id":"...","suggested_questions":[...],"candidates":[...]}`（字段按意图可为空）。
        - 异步调用 Graph。
    - **验证**：使用 Postman 或 `curl` 请求接口，能收到 Mock 的响应。

## 3. 阶段三：核心智能体开发 (Core Agents - The Brain)

**目标**：接入 LLM，让 IntentParser 和 Dispatcher 具备真实能力；同时实现多轮对话基础能力，使"澄清式导购"可验证。

**阶段验收**：输入"想买个相机"，系统能追问预算/用途；不同意图输入能正确路由到对应（Mock）Agent。

### 3.1 IntentParserAgent 实现
- [x] **Prompt 编写**
    - **内容**：在 `app/prompts/intent.py` 中编写 System Prompt，定义意图分类（Search, Outfit, QA, Chat, Compare, Plan）。
- [x] **结构化输出**
    - **内容**：使用 `pydantic` 定义 `IntentSchema`，配合 `llm.with_structured_output`。
    - **验证**：编写 10 个典型 Query 的测试用例，验证意图识别准确率 > 90%。
- [x] **槽位抽取（为澄清式导购做准备）**
    - **内容**：在意图识别结果中抽取 `budget`、`category`、`scenario`、`style`、`must_have` 等常用槽位，写入 `state.slots`。
    - **验证**：对"想买个相机""5000 元以内夜拍""男生穿搭通勤"类 Query，槽位抽取字段稳定且可复用。

### 3.2 DialogFlowAgent 实现 (多轮对话基础)
- [x] **短期上下文管理（Redis）**
    - **内容**：实现 `app/agents/dialog.py`，基于 Redis 维护会话级短期记忆。
    - **要求**：按 `thread_id` 存取对话历史；支持将历史注入 State 的 `messages` 字段。
    - **验证**：分两次请求（带相同 `thread_id`），第二次请求能引用第一次的内容。
- [x] **澄清式对话能力**
    - **内容**：当 Dispatcher 判定关键槽位缺失时，转入 DialogFlowAgent 生成追问 Prompt，返回追问响应。
    - **验证**：用户输入"想买个相机"时，系统先追问预算/用途；用户携带 `thread_id` 回答后，槽位被填充。

### 3.3 TaskDispatcherAgent 逻辑
- [x] **动态路由逻辑**
    - **内容**：完善 `dispatcher` 节点的逻辑。
    - **要求**：
        - `search` -> 转 `ShoppingAgent`
        - `outfit` -> 转 `OutfitAgent`
        - `qa` -> 转 `RAGAgent`
        - `tool` -> 转 `ToolCallAgent`
        - `chat` -> 转 `DialogFlowAgent`
        - `compare` -> 转 `ShoppingAgent`（对比模式：检索多个候选后生成对比表格）
        - `plan` -> 转 `PlannerNode`
        - `unknown` -> 回退到 `DialogFlowAgent` 进行澄清
    - **验证**：模拟不同 Intent 输入，验证图的流转路径是否正确。
- [x] **澄清式对话路由（缺槽位先追问）**
    - **内容**：当意图为 `search`/`outfit` 且关键槽位缺失（如预算/用途），由 Dispatcher 转入 `DialogFlowAgent` 生成追问；收集到槽位后再回到 `ShoppingAgent`/`OutfitAgent`。
    - **验证**：用户输入"想买个相机"时，系统先追问预算/用途；用户回答后，能进入检索路径（此阶段仍由 Mock ShoppingAgent 接收）。

## 4. 阶段四：专项智能体与数据层 (Specialist Agents - The Skills)

**目标**：接入真实数据 (ES/Milvus/MySQL)，让 ShoppingAgent、RAGAgent、OutfitAgent 具备真实业务能力。

**阶段验收**：用户完整走通"追问 -> 推荐 -> 问答"流程，返回真实商品数据。

### 4.1 数据层工具实现
- [x] **商品检索工具 (`search_products`)**
    - **内容**：封装 Elasticsearch 查询逻辑，支持 Hybrid Search（关键词 + KNN 向量）和结构化过滤（价格区间、品牌、品类）。
    - **验证**：给定"5000 元以内的微单相机"，返回符合条件的商品列表。
- [x] **知识库检索工具 (`query_knowledge`)**
    - **内容**：封装 Milvus 向量检索逻辑，支持按 `product_id` 和 `doc_type`（评论/说明书/FAQ）过滤。
    - **验证**：给定商品 ID 和问题，返回相关 chunk。
- [x] **用户画像工具 (`get_user_profile`)**
    - **内容**：封装 MySQL 查询逻辑，获取用户历史偏好（购买记录、浏览偏好、点赞/收藏）。
    - **验证**：给定 `user_id`，返回结构化用户画像。
- [x] **收藏夹工具 (`list_favorites` / `get_favorite_by_id`)**
    - **内容**：封装 MySQL 查询逻辑，支持用户列出收藏夹商品、以及在 `selected_product_id` 为空时引导用户选择。
    - **验证**：给定 `user_id` 能返回收藏列表；选择后写入 `state.selected_product_id`。

### 4.2 ShoppingAgent 实现 (导购)
- [x] **Persona 定义**
    - **内容**：编写导购专家的 System Prompt（语气热情、专业）。
- [x] **工具绑定**
    - **内容**：绑定 `search_products` 和 `get_user_profile` 工具。
    - **验证**：用户问"推荐个 5000 元的相机"，Agent 能调用工具并回答。
- [x] **个性化排序逻辑实现**
    - **内容**：基于 `get_user_profile` 返回的用户历史行为数据，实现检索结果的个性化重排序逻辑。
    - **要求**：
        - 在 `app/tools/personalization.py` 中实现排序函数，输入为 ES 检索结果列表 + 用户画像，输出为重排序后的列表。
        - 排序因子至少包含：品牌偏好、价格区间偏好、品类偏好、历史购买品类互补性。
        - 排序逻辑可插拔，便于后续替换为更复杂的模型。
    - **验证**：同一查询，不同用户画像下返回的商品排序不同；无画像用户退化为默认排序。
- [x] **候选集输出结构化（满足"多个产品供选择"）**
    - **内容**：定义候选商品输出结构（如 `candidates: [{product_id,title,price,reason,pros_cons}]`），并在自然语言回答中引用。
    - **验证**：同一查询能稳定返回 \(N\ge 3\) 个候选商品；每个候选包含可解释理由且不编造未检索到的参数。

### 4.3 RAGAgent 实现 (问答)
- [x] **RAG 流程构建**
    - **内容**：实现 `app/agents/rag.py`。
    - **要求**：接收 Query -> 调用 `query_knowledge` -> 上下文注入 Prompt -> 生成回答。
    - **验证**：问"这款相机夜拍怎么样"，能基于 Milvus 中的评论回答。
- [x] **收藏夹商品深度问答（对齐需求）**
    - **内容**：当 `selected_product_id` 存在时，RAG 默认带 `product_id` 过滤；当不存在时，引导用户从收藏夹选择或让用户提供商品范围。
    - **验证**：用户先选择收藏夹商品，再提问参数/口碑时，回答能引用该商品相关 chunk，避免跨商品串答。

### 4.4 ToolCallAgent 实现 (执行)
- [x] **通用工具执行器**
    - **内容**：实现 `app/agents/tool_call.py`，用于处理非对话类的纯工具调用任务（如查订单状态、物流查询）。
    - **验证**：测试调用特定工具函数，返回正确结果。

### 4.5 OutfitAgent 实现（穿搭/组合推荐）
- [x] **OutfitAgent 节点**
    - **内容**：新增 `app/agents/outfit.py`，将"男生穿搭"等跨品类需求拆解为上衣/下装/鞋/配饰等组合清单，可复用 `search_products` 进行多次检索并整合。
    - **输出**：结构化输出全身搭配（包含品类、价位区间、风格说明、可替代项）。
    - **验证**：用户输入"男生穿搭（通勤/约会/休闲）"，返回全身搭配清单且每个单品可点选/可替换。

### 4.6 ResponseFormatter 正式实现
- [x] **响应润色与格式化节点**
    - **内容**：完善 `app/agents/response_formatter.py`（替换阶段二的 Mock），统一处理各 Agent 输出的语言润色与格式化。
    - **要求**：
        - 将各 Agent 的结构化输出（候选商品列表、穿搭清单、RAG 回答等）转换为统一的用户友好格式。
        - 确保不同 Agent 的输出风格一致（语气、排版、信息密度）。
        - 拼装最终 Response Body 的各字段（`response`, `candidates`, `suggested_questions` 等）。
    - **验证**：不同类型的查询（导购/问答/穿搭）返回的响应风格统一，结构完整。

### 4.7 相关问题推荐（Proactive Engagement）
- [x] **`suggested_questions` 生成**
    - **内容**：在 ResponseFormatter 中增加相关问题生成逻辑（规则 + LLM 皆可），用于引导用户继续提问。
    - **验证**：任意一次回答后，返回 3-5 个不重复、与当前主题强相关的问题。

## 5. 阶段五：高阶推理能力 (Advanced Reasoning)

**目标**：实现 Reflection (反思) 和 Planning (规划) 能力，提升 Agent 智能度与鲁棒性。

**阶段验收**："200元买全新单反"能被识别为不合理并引导用户；"西藏旅游全套装备"能被拆解为多步子任务执行。

### 5.1 Reflection & Self-Correction (反思)
- [x] **Reflector Node 实现**
    - **内容**：在 `ShoppingAgent` / `OutfitAgent` 后增加 `Reflector` 节点。
    - **逻辑**：检查检索结果是否为空？回答是否符合用户约束？若不符合，生成修正建议并回退给上游 Agent 重试。
    - **要求**：
        - 明确循环控制：最大重试次数（建议 ≤ 3）。
        - 每次重试的策略变化：放宽/收紧过滤 -> 改写 query -> 请求澄清。
        - 失败出口：转澄清（DialogFlowAgent）/ 给出可执行建议（如"您的预算建议调整到 XXX"）。
    - **验证**：模拟"200元买全新单反"的查询，验证 Agent 是否能识别不合理需求并引导用户。

### 5.2 Task Planning (规划)
- [x] **Planner Node 实现**
    - **内容**：针对复杂 Query (如"西藏旅游全套装备")，引入 `app/agents/planner.py`。
    - **逻辑**：使用 ReAct 或 COT 思维链，将大目标拆解为子任务序列 (查天气 -> 选衣物 -> 选药品 -> 选器材)。
    - **要求**：
        - Plan 必须可执行（每步绑定到具体 Agent/Tool），输出为 `[{step, agent, params}]` 结构。
        - 具备中止条件与错误处理（某步检索为空时回退到澄清或调整检索策略）。
        - Dispatcher 在识别到 `plan` 意图时路由到此节点。
    - **验证**：输入复杂任务，检查生成的 Plan 是否合理，以及子任务执行顺序和最终整合结果。

## 6. 阶段六：可靠性与记忆增强 (Reliability & Memory)

**目标**：完善长期记忆、模型智能路由与调度可靠性，使系统具备生产级鲁棒性。

**阶段验收**：跨会话偏好记忆生效；主模型故障时自动降级且业务不中断；并发请求下系统稳定。

### 6.1 长期记忆与记忆迁移
- [x] **记忆迁移策略（短期 -> 长期）**
    - **内容**：定义 Redis 短期记忆向 Milvus 长期记忆迁移的策略与触发时机。
    - **要求**：
        - 会话结束（用户显式结束或超时）时，将对话摘要向量化后写入 Milvus，关联 `user_id`。
        - 定义 Redis 过期策略（如 TTL 30 分钟），过期前触发迁移。
        - 新会话开启时，从 Milvus 检索该用户近期对话摘要，注入上下文实现跨会话记忆。
    - **验证**：用户在会话 A 中表达"喜欢复古风"，会话 B 中询问穿搭时，系统能引用该偏好。

### 6.2 FallbackAgent 实现 (智能模型路由)
- [x] **模型智能路由与降级机制**
    - **内容**：实现 `app/agents/fallback.py`，将阶段一的基础降级逻辑升级为智能路由。
    - **要求**：
        - 支持基于任务复杂度/成本的模型路由（如简单澄清/改写用轻量模型，复杂规划/反思用强模型）。
        - 主模型调用失败时自动降级至备用模型，记录路由决策与失败日志。
        - 监控模型接口状态（响应时间、错误率），动态调整路由策略。
    - **验证**：不同复杂度任务使用不同模型；主模型不可用时自动切换且对用户透明。

### 6.3 调度可靠性
- [x] **超时/重试/幂等策略**
    - **内容**：为 LLM 调用与工具调用定义统一超时、重试次数、退避策略；对同一 `trace_id + step` 做幂等保护（避免重复写入/重复扣费）。
    - **验证**：模拟工具超时/LLM 超时场景，系统能降级或返回可解释错误，不阻塞整个请求。
- [x] **并发调度边界**
    - **内容**：定义可并发的子任务（如 Outfit 多品类检索），限制最大并发数，避免压垮 ES/Milvus。
    - **验证**：压力测试下 ES/Milvus 连接数稳定，响应延迟可控。
