# 开发计划 (Development Roadmap)

基于已有的数据层（MySQL, ES, Milvus），本项目将按照以下四个阶段进行开发。核心策略是 **“工具先行，大脑组装，最后产品化”**。

## 阶段一：工具封装与检索层 (Tooling & Retrieval Layer)
**目标**：把“死数据”变成 Agent 可以调用的“活技能”。LLM 不直接写 SQL 或 DSL，而是调用 Python 函数。

### 1.1 定义核心工具 (Tools Definition)

我们需要为 Agent 准备以下几类工具，覆盖从商品检索到个性化推荐的全流程：

#### A. 核心检索类 (Core Retrieval)
1.  **商品检索工具 (`search_products`)**
    *   **对接组件**: Elasticsearch
    *   **功能**: 实现 **Hybrid Search**（关键词匹配 + 向量相似度），并支持个性化重排序。
    *   **输入参数**:
        *   `query` (str): 用户搜索词（如“红色连衣裙”）
        *   `price_min` / `price_max` (float): 价格区间
        *   `brand` (str): 品牌过滤
        *   `category` (str): 品类过滤
        *   `sort` (str): 排序方式（default/price_asc/price_desc/sales/rating）
        *   `user_preferences` (list[str]): **[新增]** 用户偏好标签列表（如 `["复古", "极简"]`），用于 Function Score 加权排序。
    *   **关键点**: 确保能同时处理语义理解（“复古风”）和硬性过滤（“500元以下”），并根据用户偏好提升相关商品的排名。

2.  **知识库问答工具 (`query_product_knowledge`)**
    *   **对接组件**: Milvus
    *   **功能**: 检索非结构化文本（评论、说明书、FAQ）。
    *   **输入参数**:
        *   `product_id` (str): 目标商品 ID
        *   `query` (str): 用户具体问题（如“这款相机夜拍噪点大吗？”）
        *   `doc_type` (list[str]): **[新增]** 文档类型过滤，可选 `["review", "spec"]`。
            *   *Scenario*: 问参数查 `spec`，问体验查 `review`。
    *   **逻辑**: 先根据 `product_id` 和 `doc_type` 过滤，再对 `query` 进行 Embedding，检索 Top-K 相关 Chunks。

3.  **多商品对比工具 (`compare_products`) [新增]**
    *   **对接组件**: Elasticsearch
    *   **功能**: 获取多个商品的结构化参数进行横向对比。
    *   **输入参数**: `product_ids` (list[str])
    *   **输出**: 结构化的对比表格数据（如：像素、重量、价格对比）。

#### B. 用户与个性化类 (User & Personalization)
4.  **用户画像工具 (`get_user_profile`)**
    *   **对接组件**: MySQL
    *   **功能**: 获取用户历史行为数据。
    *   **输入参数**: `user_id`
    *   **输出**: 历史订单列表、最近浏览记录、偏好标签（用于传给 `search_products`）。

5.  **收藏夹管理工具 (`manage_favorites`)**
    *   **对接组件**: MySQL / Redis
    *   **功能**: 管理用户的收藏夹（添加/删除/查询）。
    *   **输入参数**: `action` (add/remove/list), `product_id`

#### C. 辅助与增强类 (Auxiliary & Enhancement)
6.  **趋势推荐工具 (`get_trending_products`)**
    *   **对接组件**: Redis / MySQL
    *   **功能**: 获取当前热门商品或榜单。
    *   **输入参数**: `category` (可选), `top_k` (默认 5)
    *   **用途**: 当用户没有明确需求（“随便看看”）时使用。

7.  **相似商品推荐工具 (`get_similar_products`)**
    *   **对接组件**: Milvus / ES
    *   **功能**: 基于当前查看的商品，推荐“猜你喜欢”。
    *   **输入参数**: `product_id`
    *   **逻辑**: 基于商品向量寻找最近邻。

### 1.2 工具测试 (Unit Testing)
*   **DoD (完成标准)**: 编写 `test_tools.py`，不依赖 LLM，直接调用 Python 函数能查到准确数据。
    *   *Case*: 调用 `search_products(price_max=2000, user_preferences=["复古"])` 必须返回 2000 元以下的商品，且复古风格的商品排名靠前。

## 阶段二：构建基础 Agent (The MVP Agent)
**目标**：实现一个能听懂话、能找货的“实习导购”。利用 LangGraph 组装基础循环。

### 2.1 状态管理 (State Management)
*   定义 `AgentState`，包含：
    *   `messages`: 对话历史
    *   `user_intent`: 当前意图分类
    *   `search_params`: 提取出的结构化搜索参数
    *   `retrieved_items`: 召回的商品列表

### 2.2 意图识别路由 (Router)
*   实现分类器节点，判断用户意图：
    *   **Search**: “我想买...” -> 路由到商品检索 Agent。
    *   **QA**: “这个支持5G吗？” -> 路由到 RAG Agent。
    *   **Compare**: “对比一下这两款” -> 路由到对比 Agent。
    *   **Chat**: “你好” -> 直接回复。

### 2.3 参数提取 (Slot Filling)
*   利用 LLM 从自然语言中提取结构化参数。
    *   *Example*: “两千左右的耐克鞋” -> `{"brand": "Nike", "price_min": 1800, "price_max": 2200}`

### 2.4 DoD (完成标准)
*   在控制台（CLI）中，用户输入“推荐个两千块的手机”，Agent 能调用 ES 工具并返回商品列表文本。

## 阶段三：赋予“高级智能” (Advanced Intelligence)
**目标**：实现 **反思 (Reflection)** 和 **规划 (Planning)**，解决“人工智障”和“幻觉”问题。

### 3.1 反思机制 (Self-Reflection Loop)
*   **场景**: 用户搜索“500元以下的全新 iPhone 15”（条件过于苛刻）。
*   **逻辑**:
    1.  Agent 调用搜索工具 -> 结果为 0。
    2.  进入 **Reflection Node** -> 分析原因（预算不合理）。
    3.  **修正策略** -> 自动调整搜索条件（去掉“全新”或提高预算）或生成反问引导用户。
    4.  再次搜索。

### 3.2 复杂任务规划 (Planner)
*   **场景**: “我要去西藏旅游，帮我准备装备”。
*   **逻辑**:
    1.  **Planner Node** 将大目标拆解为子任务序列：`[查天气, 选冲锋衣, 选氧气瓶, 选高热量食物]`。
    2.  Agent 依次执行搜索，最后汇总推荐清单（Bundle Recommendation）。

### 3.3 DoD (完成标准)
*   搜不到结果时，Agent 不会幻觉，而是会尝试修正或解释。
*   面对复杂场景，能给出组合式建议。

## 阶段四：接口服务与评测 (Service & Evaluation)
**目标**：从 Demo 变成 Product，并量化质量。

### 4.1 API 服务化 (FastAPI)
*   封装 `/chat` 接口。
*   支持流式输出 (Streaming Response)。
*   集成 Redis 存储 Session History，支持多轮对话记忆。

### 4.2 自动化评测 (Ragas)
*   构建测试集 (Golden Dataset)：准备 50 个典型 Query 和标准答案。
*   运行 Ragas 评估：
    *   **Faithfulness**: 回答是否忠于检索到的商品信息。
    *   **Answer Relevance**: 推荐是否符合用户需求。

### 4.3 DoD (完成标准)
*   前端界面可交互（Streamlit 或 React）。
*   产出一份自动化的质量评测报告。
